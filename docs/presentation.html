01 · Presentation Summary

One engineer, five subagents, one shipped module.

The DMS Vehicle Inventory module — authentication, a KPI dashboard, full vehicle CRUD, per-column filtering, and pagination — was designed and shipped end-to-end using Claude Code as the development harness. In place of a team, the project ran on a written constitution (CLAUDE.md), five purpose-built subagents, four skills, and three commands, all enforcing one non-negotiable rule: components never touch Supabase directly, business logic lives in services, and repositories are the only door to the database.

02 · Architecture

One Next.js app, split by hard import boundaries, not by repo.

"Frontend/backend separation" here doesn't mean two deployments — it means an ESLint rule that fails the build if a page imports Supabase directly. Everything ships together on Vercel; the boundary lives inside the app.

Why Next.js

App Router gives one framework for pages, API routes, and Server Actions — no separate backend service to stand up or deploy. That's the right amount of infrastructure for a small-to-medium DMS module: fast to build, easy to reason about, and TypeScript end-to-end from a Postgres row to a rendered table.

Why Vercel

Vercel is built by the Next.js team for exactly this shape of app: push a branch, get a preview deployment automatically, merge to main, get production — no servers to provision, no separate CI-to-hosting handoff to wire up.

03 · Plan to Attack the Solution

Ten steps, in the order they actually shipped.

The build followed a deliberate sequence: get the harness right before writing feature code, get auth right before writing data screens, get the table right before layering filters and pagination on top of it.

Step 04

Added the ESLint FE/BE boundary rule, the GitHub Actions CI pipeline, and the first draft of commands, skills, and agents for feature work.

Step 05

Auth repository and service, login/signup Server Actions, session-refresh middleware, and a protected dashboard shell. PR #2

Step 06

Vehicle inventory schema migration with seed data, repository/service layer, and the KPI-summary dashboard UI. PR #3

Step 07

Add CRUD functionality

Repository/service CRUD, create/edit/delete UI, a detail page, and real-time form validation gating submit. PR #4

Step 08

Add search and filtering

Per-column filters on the vehicle list query, consolidated into a single filter stack component. PR #9

Step 09

Add pagination

Paginated the inventory list, then fixed out-of-range page requests to clamp to the last page instead of returning empty. PR #10

Step 10

Verify the deployed app is stable

Every merge passed lint, format check, and Vitest in CI, plus a Playwright browser smoke check from the qa agent before push.

05 · AI Development Workflow

Agents, skills, and commands — and how they talk to each other.

All three sit on top of one shared contract, CLAUDE.md. Commands are explicit, slash-invoked shortcuts for a single scoped task. Skills are implementation patterns — pulled in automatically when relevant, or named directly — that keep output consistent without re-explaining conventions every time. Agents are autonomous, multi-step workers that carry out a full task end to end, applying the same skills and conventions a human would.

Subagents

build-feature Day 1

Implements a full vertical slice — repository, service, action/route, UI — while respecting the FE/BE boundary.

Skills

scaffold-crud

Pattern for generating repository, service, and Server Action layers for a new domain entity — the CRUD five: find all, find by id, create, update, remove.

react-form

Pattern for a form Client Component wired to a Server Action, with real-time client-side validation that mirrors — never replaces — service-layer validation.

api-endpoint

Pattern for a thin API route: parse input, call exactly one service function, return an explicit response — for the cases a Server Action doesn't fit.

design-system

The visual language for every UI change in this repo — color, type, spacing, radius, motion. Extracted directly from an HTML design reference the user shared, then codified into reusable tokens.

06 · Two-Day Development Workflow

Day one builds it. Day two proves it.

The harness splits the lifecycle across two agent rosters: builders on day one, verifiers on day two. A feature isn't done when it compiles — it's done when a reviewer agent and a browser agent have both signed off.

Pagination shipped on day one. Day two's verification surfaced a real edge case — a page number past the last page returned an empty list instead of clamping to it. bug-triage reproduced it, traced it to the service-layer normalization, fixed it, and added both a unit and an e2e regression test before the branch was allowed to merge.
